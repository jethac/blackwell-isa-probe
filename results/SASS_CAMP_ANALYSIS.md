# SASS camp analysis — sm100 (datacenter) vs sm120 / sm121 (consumer)

Every probe compiled for `sm_100a`, `sm_120a`, `sm_121a` on `ptxas` V13.3.73,
disassembled (`cuobjdump`), machine code diffed across the three. The compile
matrix answers "does it assemble"; this answers "does it lower to the same
**mechanism** on datacenter vs consumer." Set = 993 documented-PTX + 23 PTX-9.3
delta = 1,016. **Rigorous harness: `harness/sass_camp_analysis2.py`** (sink-wrapped
+ marker-classified — see §D); raw: `results/sass_camp_report2.json`.

## Headline
- **815 no-marker-diff** — the compute ISA is common across camps; **sm120 ≡ sm121**
  everywhere (zero intra-consumer differences).
- **20 mechanism splits** — consumer traps / software-emulates / degrades a
  datacenter hardware op (§B). **13 native-encoding differences** — both camps do
  it natively via a different instruction (§C).
- **168 unsinkable**, broken down: **155 camp-exclusive / off-spec** (bare-reject
  on ≥1 target — these *are* §A's compile-level splits + ~68 negative-test probes;
  no 3-way SASS comparison exists), **3 raw** hand-written kernels, and **10
  genuine sink-wrapper gaps** (bare compiles on all three, the generic wrapper
  broke). The 10 gaps = 4 `bra` control-flow (no data result to compare) + 6 fp4
  packed converts (`cvt e2m1x2 ↔ f16x2/bf16x2/f32`; the wrapper lacks a rule for
  the `.b8`/`.f16x2` packed operands). **The 6 were hand-sunk and verified
  camp-identical** (`F2FP.*.E2M1.PACK/UNPACK` same on both camps) — so no hidden
  split escapes. The wrapper could auto-cover them by adding packed-type
  handling; the question is already closed by hand.

## A. Compile-level split (one camp rejects outright)
*(Exact counts machine-generated by `results/summarize_camp.py` → `CAMP_SPLIT.md`;
strict "all-datacenter vs all-consumer" membership.)*
- **68 consumer-only** — the fp4/fp6 block-scaled & dense `mma.sync` family
  (`kind::mxf4nvf4` / `mxf4` / `mxf8f6f4` / `f8f6f4`, 52) + the saturating `u8x4` /
  `s8x4` / `s16x2` packed-SIMD integer ops (16). Reject on all datacenter.
- **7 datacenter-only** — `tcgen05.*` (alloc/ld/cp/commit/mma) + paired-CTA
  `cp.async.bulk…cta_group::2`. Reject on all consumer.
- **11 datacenter-*subset*** — `cvt.rs.*` (stochastic rounding, 7) +
  `redux.sync.{min,max}.f32` (4): accept on **B200/B300 only, not Thor**, and
  reject on all consumer. (The finer sub-split a blunt "datacenter-only" hides.)

## B. SASS mechanism splits (20) — consumer traps / emulates / degrades
Assemble on all three, but the machine code shows a capability marker on one camp
the other lacks. Each sink-verified.

| Family | n | Datacenter (sm_100a) | Consumer (sm_120a/121a) |
|---|---|---|---|
| cluster async-copy `cp.async.bulk{.tensor.1-5d,.im2col}.…shared::cluster` (+`.multicast`) | 9 | pure hardware TMA (`UBLKCP`) | remote-rank path → `SR_CgaCtaId` → `LEPC` → **`CALL`** (software trampoline) |
| `fabric.try_{get,put,red}` | 4 | real path (`UBLKCP`) | **`BPT.TRAP`** — non-functional stub |
| async `multimem.st/red.async`, `red.async.sys` | 4 | true async (`UREDGR`) | async dropped → `ERRBAR`+`CGAERRBAR`+**synchronous `REDG`** |
| sparse fp8 MMA `mma.sp…e4m3` | 3 | **`CALL`**+`HMMA.SP.16832`×2 (helper path) | one native `QMMA.SP.16864` |

**Through-line:** cross-CTA / multi-GPU / sparse-tensor ops are dedicated hardware
on the tcgen05 camp and are **software-emulated, trapped, or degraded** on
consumer — invisibly to the compile oracle. Guard by camp, not by compile-accept.

## C. SASS native-encoding differences (13) — both native, different instruction
Real codegen differences, but neither camp is emulating — worth knowing (perf),
not a capability gap.
- **plain fp8 `mma.sync` (10)** — datacenter lowers to `HMMA`, consumer to native
  `QMMA`. Both do fp8 MMA; different tensor-core instruction lineage.
- **`redux.sync.{min,max}.u32` / `red.global.min` (3)** — datacenter `CREDUX`,
  consumer `REDUX`. Both do the warp reduce; different encoding.

## D. Method — how the three v1 failure modes were fixed
A naive opcode-set diff over *bare* compile-probes fails three ways (all
demonstrable on trivial ops): (1) DCE blind spot — `add.s32` with unused output
vanishes; (2) undefined-input scaffolding — `rcp` shows `VIADD`, `atom.min` shows
`CREDUX`, both spurious; (3) selection noise — sink-wrapped `add.s32` is
`IADD3` (dc) vs `IADD` (consumer), same op. `sass_camp_analysis2.py` fixes all
three:
- **Sink-wrap every probe** — every register the op reads is initialized from a
  `volatile` global load (defined, non-constant); the destination is stored. No
  DCE, no undefined inputs. (Side-effecting ops need no dest store.)
- **Classify by capability marker, not opcode set** — a split registers only when
  a *mechanism* marker differs (`BPT`, `CALL` trampoline, `ERRBAR`/`CGAERRBAR`
  guard) — §B — or a *tensor/reduce instruction class* differs (§C, reported
  separately). `IADD3`≡`IADD` and other benign selection carry no marker.

Validation: `add.s32`, `rcp.rn.f32`, `atom.global.min` → **no-marker-diff**;
`fabric.try_get`, cluster-copy, sparse-MMA → **camp-split**. The v1 hand-pruned
result, reproduced automatically and exhaustively. (v1 harness
`sass_camp_analysis.py` + `sass_camp_report.json` retained for comparison.)

## E. ptxas 13.3.73 bugs (no new ICEs in this sweep)
- `clmad.{lo,hi}.u64` — miscompiled on all four Blackwell targets (accepts, emits
  no carry-less arithmetic, stores the wrong operand; spec §9.7.1.5 = `clmul(a,b)+c`).
- `red.async.release.sys…add.u32 [a],v,[mbar]` (mbarrier form) — C7907 internal
  compiler error / segfault, datacenter and consumer alike.
